[react-native-worklets](https://docs.swmansion.com/react-native-worklets/docs/fundamentals/getting-started/)
(Software Mansion) and React Native Runtimes both run JavaScript off the main
runtime, but they solve different problems with different execution models.

## At a glance

| # | Aspect | react-native-worklets | react-native-runtimes |
| --- | --- | --- | --- |
| 1 | **What runs on the other thread** | Short-running functions marked `'worklet'`. A Babel plugin "workletizes" them â€” extracts the function and its captured closure into a serializable object that is *copied* onto the worklet runtime. | Ordinary JS â€” full React component trees and whole screens. Each runtime has access to **all core React Native components** (plus any registered native module), which is what makes it possible to render complete UI on a background thread. |
| 2 | **Rendering & shadow tree** | âťŚ Worklet runtimes can't mount React trees. UI-thread worklets (Reanimated) mutate the **main tree's ShadowNodes**, so they are subject to main-runtime commit locks. | âś… [`<OnRuntime>` / `ThreadedScreen`](/docs/threaded-runtime/render-components) run a real React renderer per runtime with **its own shadow tree â€” no shared ShadowNodes**, so background runtimes are **never blocked by main-runtime locks**. |
| 3 | **Bundle & module access** | By default worklets only see their workletized closure â€” third-party imports need patches. Full-bundle access requires the **experimental [Bundle Mode](https://docs.swmansion.com/react-native-worklets/docs/bundleMode)** (`react-native-worklets@bundle-mode-preview`, Babel `bundleMode: true`, temporary Metro patches). Even then, third-party libraries must be allowlisted via `workletizableModules`, and `fetch` requires a separate feature flag. | Full bundle access **is the default and only mode** â€” Metro generates `.threaded-runtime/entry.js`, every runtime loads a complete bundle, so any JS import or third-party library just works. **Native modules need a one-time registration** in `MainApplication.kt` â€” [`setExtraReactPackagesProvider`](/docs/getting-started/android-setup) for threaded UI runtimes, `setMainReactPackagesProvider` for business runtimes. |
| 4 | **App startup** | Worklet runtimes are created from JS (`createWorkletRuntime`) **after** the main bundle has loaded â€” startup work stays serialized behind the main runtime. | Secondary runtimes can be **started when the app process starts** ([`prewarmBusinessRuntime`](/docs/threaded-runtime/business-logic-executor) / [`prewarm`](/docs/threaded-runtime/prewarming)), so app initialization, hydration, and first-screen prep run **in parallel with main-runtime startup**. |
| 5 | **Cross-thread state & communication** | Shared values / Shareables plus `scheduleOnUI` / `scheduleOnRN` (formerly `runOnJS`, now deprecated); captured data is copied (serialized) between runtimes per call. | JSON-serialized props/args, typed [`call(fn).on('runtime')`](/docs/threaded-runtime/scheduling-functions), [headless tasks](/docs/threaded-runtime/headless-runtime), plus [`@react-native-runtimes/state`](/docs/shared-state/overview) â€” a C++-singleton-backed Zustand-style store with **synchronous reads/writes from every runtime**. |

<Callout type="info" title="Complementary, not competing">
  Worklets win for synchronous, frame-precise work on the UI thread â€”
  animations and gestures, where Reanimated is built on them. React Native
  Runtimes wins for moving entire screens, React trees, and app-startup or
  business logic off the main JS thread. Many apps benefit from using both.
</Callout>
